# 채점 기준 & 게이팅 규칙

**활성 축만** 채점한다 (활성 여부는 [repo-types.md](repo-types.md)의 매트릭스로 결정). 각 활성 축은 0–100점, 항목 점수의 가중 합으로 축 점수를 낸다. 비활성 축은 `N/A`로 표기하고 채점·게이팅·평균에서 제외한다. 모든 항목은 증거(파일 경로/grep/실행 로그)로 뒷받침되어야 하며, 증거 없는 항목은 만점을 줄 수 없다.

축 상태는 `pass | concern | fail | n/a` 중 하나로 표기한다.

## 축① 기획의도 정합성 (intent conformance)

| 항목 | 가중 | 만점 기준 |
|---|---:|---|
| 범위 추적성 | 40 | 선언된 기능/MVP 항목이 모두 `구현됨`이고 각각 구현 증거 경로가 있음 |
| 부분/미구현 비율 | 20 | `부분구현`·`미구현` 항목이 적고, 미구현은 문서상 의도된 후속 작업으로 명시됨 |
| 제약 준수 (게이팅) | 40 | "제외" 항목과 Do-Not 규칙 위반이 0건 (grep 역검증) |

- **상태 정의**: `구현됨`(증거 명확) / `부분구현`(일부 경로만 존재·미완) / `미구현`(증거 없음) / `확인필요`(증거가 침묵·상충해 의도 일치 판독 불가).
- **`확인필요` 처리**: 범위 추적성 채점의 분모에서 제외(pass/fail 아님). FAIL/CONCERNS를 직접 트리거하지 않으나, **핵심 범위 항목에 1건 이상 있으면 축① 상태는 pass 불가(상한 concern)** — 미검증을 통과로 위장하지 않는다. `--interview`로 해소된 항목은 확정 상태로 정상 채점한다.
- **게이팅**: 제약 위반 1건 이상 발견 시 → 위반당 Critical 1건, 축① 점수 상한 40, 전체 판정 FAIL.

## 축② AI-readable 구조 + 문서 완전성

| 항목 | 가중 | 만점 기준 |
|---|---:|---|
| 에이전트 가이드 | 20 | `AGENTS.md` 또는 `CLAUDE.md`가 범위·명령·DoD·제약을 명시 |
| README 품질 | 15 | 목적·워크플로·주요 디렉토리·실행 명령이 기술됨 |
| 구조 일관성 | 20 | 예측 가능한 디렉토리 분리, 과도한 중첩 없음, 일관된 네이밍 |
| 기계 판독 데이터 | 15 | 스키마/설정이 구조화 포맷(JSON 등)이며 검증 스크립트 존재 |
| 문서-코드 정합성 | 15 | 문서에 적힌 명령·경로가 실제로 존재(불일치는 Major) |
| 문서 완전성 | 15 | 타입에 필요한 문서 존재 — 라이브러리/서비스면 LICENSE·CONTRIBUTING·CHANGELOG, 모든 타입 공통 README. 타입에 불필요한 문서 부재는 감점 안 함 |

## 축③ goal 검증 + 테스트 충분성 (verification)

| 항목 | 가중 | 만점 기준 |
|---|---:|---|
| 테스트 실행 (게이팅) | 30 | 테스트 명령이 존재하고 실제 실행 시 exit 0 |
| validator 실행 | 20 | 스키마/lint validator가 대상 전체에 대해 통과 |
| 테스트 충분성 | 20 | 테스트가 핵심 기능·성공지표를 실제로 커버. 빈/형식적/항상-통과 테스트가 아니며, 커버리지 도구가 있으면 그 수치를 인용 |
| 성공지표/DoD 충족 | 20 | 기준 문서의 성공 지표·DoD가 증거와 함께 충족됨 |
| 증거 기록 | 10 | 위 결과가 실행 로그(stdout/exit)로 캡처됨 |

- **게이팅**: 테스트 실패(exit≠0) 또는 핵심 DoD 미충족 시 축③ 점수 상한 50, 전체 판정 FAIL. 테스트가 존재하나 핵심 기능을 전혀 커버하지 않으면 Major.

## 조건부 모듈 (Phase 1에서 활성화된 것만 채점)

각 모듈도 0–100점. 비활성이면 `N/A`. `필수` 표시 모듈의 치명 결함은 Critical → 전체 FAIL.

### M-SEC 보안/시크릿 위생

| 항목 | 가중 | 만점 기준 |
|---|---:|---|
| 하드코딩 시크릿 (게이팅) | 60 | 키/토큰/비밀번호/프라이빗키 grep 결과 0건 (`api[_-]?key`, `secret`, `password`, `BEGIN PRIVATE KEY`, AWS 키 패턴) |
| 민감 파일 추적 | 25 | `.env`·credential 파일이 커밋되지 않고 `.gitignore`로 보호됨 |
| 보안 정책 | 15 | 타입상 필요하면 `SECURITY.md` 존재 (불필요하면 N/A 처리) |

> **리포트 위생**: 탐지한 시크릿을 리포트에 적을 때는 **값을 마스킹**(앞 2~4자 + `***`)하고 위치(`path:line`)만 기록한다. 평문 값을 본문에 옮기면 노출 범위가 원본보다 넓어진다.

## Severity 정의

- **Critical**: 제약 위반, 테스트 실패, 선언 핵심 기능 미구현. → 전체 FAIL 트리거.
- **Major**: 문서-코드 불일치, 부분구현, DoD 부분 미충족. → 다수 시 CONCERNS.
- **Minor**: 네이밍·문서 보강 등 품질 개선 제안.

## 종합 판정 (verdict)

판정은 **활성 축의 발견사항에 한해** 적용한다. 비활성(N/A) 축은 어떤 판정도 트리거하지 않는다.

| 판정 | 조건 |
|---|---|
| **FAIL** | 활성 축 중 Critical ≥ 1 (제약 위반·테스트 실패·필수 모듈 치명 결함 포함) |
| **CONCERNS** | Critical 0 이고 Major ≥ 2 |
| **PASS** | Critical 0 이고 Major ≤ 1 |

전체 점수는 **활성 축 평균**(참고용). 판정은 점수가 아니라 위 게이팅 규칙으로 결정한다.

## Fallback (기획 문서 부재 시)

기준 문서를 못 찾으면 A1은 README/AGENTS.md에 적힌 목표·기능 진술을 baseline으로 대체하고, baseline 부재 자체를 Major로 기록한다. 다른 활성 축은 그대로 적용한다.
